Site Notes

About This Site

This site was built specifically for my application to UCSC for Fall 2026. It's also an example of the kind of work I do: designing for specific audiences, making deliberate choices about audience and purpose, accessibility and presentation, and then shipping functional systems without unnecessary complexity.

Design

The visual language borrows from UCSC's brand identity: navy (#003C6C) and gold (#F1B521), while extending it with lighter variants for backgrounds and accents. I avoided dense bullet lists and heavy formatting in favor of prose that reads more like a conversation and less like a slide deck.

Typography

The primary typeface is Readex Pro, the world-script expansion of Lexend. Lexend was designed by Bonnie Shaver-Troup and Thomas Jockin specifically to improve reading fluency, and studies found it helps readers with dyslexia and reduces visual stress for everyone. Readex Pro extends that work to Arabic script (added by Nadine Chahine) while maintaining the clean, geometric letterforms that make it readable at any size. It also renders Hawaiian diacritical marks correctly—the ʻokina and kahakō that are essential to Hawaiian language. Given my work with the Hawaiian language community, that matters to me.

I chose Readex Pro not just for aesthetics but because accessibility research backs it. For a portfolio about technology in service of community, I thought the typeface should embody that principle.

Source Code Pro handles the few monospace elements I used, such as the breadcrumb navigation.

Accessibility

The site targets WCAG AAA compliance where possible. This includes: color contrast ratios above 7:1 for body text, skip links for keyboard navigation, semantic HTML structure, and alt text for images. I tested with WAVE and keyboard-only navigation. The gold-on-navy and navy-on-gold combinations were specifically chosen to pass contrast requirements.

Technical

Static HTML with Tailwind CSS (via CDN) and Lucide icons, hosted on GitHub Pages. No local build step, no bundler, no framework beyond utility CSS. This was intentional, because I wanted something I could edit directly and that would remain stable with minimal fuss. GitHub Pages seemed like the right choice, as the simplicity of it and direct access to the code is, well... kind of the point.

Process

I built this site with AI assistance, and I want to be transparent about what that means.

AI tools helped with CSS implementation, accessibility testing, and iterating through revisions faster than I could have alone. Claude Code handled much of the line-by-line build process; other AI tools provided external code review (I rarely work with a single LLM exclusively on any given project because they all have their strengths and drawbacks). When something didn't work or didn't look right, I made direct manual corrections. But the design decisions, structure, editorial voice, and PRs were all mine—not because I wrote every line from scratch, but because I reviewed, revised, and iterated as needed and I take responsibility for what's here.

That 'responsible use' distinction matters to me. In the era we've entered, the question isn't whether AI was involved in producing something; it's whether a human took responsibility for the final output. Simon Willison puts it simply: your job is to deliver code you have proven to work. That's the standard I tried to meet.

AI couldn't decide what was important about my work. It couldn't know that following UCSC's brand colors while departing from the specified typeface (Roboto) in favor of one designed for reading fluency (Readex Pro) was a reasonable trade-off for this particular audience and context. Those choices required understanding what I was trying to communicate, and to whom.

That's the kind of judgment I don't think AI should replace., Even if we thought it could.
